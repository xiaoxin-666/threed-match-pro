# 3D Match Pro 改进建议

基于当前版本（1.1）的全面审计，以下是按优先级排列的改进建议。

---

## 一、高优先级（影响大、实现快）

### 1. 请求失败自动重试

**现状**：单次 HTTP 错误或网络抖动就导致整个任务停止（状态 ERROR），没有重试。

**建议**：在 `ExecutionEngine` 中增加重试逻辑——单个请求失败后自动重试 3 次（间隔 2s/4s/8s 递增），全部失败才标记 ERROR。

### 2. 已完成任务增加"重新运行"按钮

**现状**：任务完成后状态为 COMPLETED，play 按钮不可用。"启动全部"也跳过已完成任务。用户想重新跑只能手动编辑。

**建议**：任务卡片上为 COMPLETED 状态增加"重新运行"按钮（循环箭头图标），点击后重置 `completedCount = 0`、`status = PENDING` 并自动启动。

### 3. 控制台日志按级别过滤

**现状**：所有日志（INFO/SUCCESS/ERROR）混在一起，长时间运行后找错误很困难。

**建议**：控制台顶部增加一排过滤按钮：[全部] [信息] [成功] [错误]，点击后在 `LazyColumn` 中仅显示对应级别的日志。

### 4. 商品查询走代理

**现状**：`GoodsInfoApi` 用独立 OkHttp 客户端，不走用户配置的代理。开启代理后查询行为与任务行为不一致。

**建议**：让 `GoodsInfoApi` 接收 `ProxyConfig` 参数，复用 `createOkHttpClient(proxyConfig)`，与 admire 请求使用同一出口 IP。

---

## 二、中优先级（体验提升）

### 5. 合并重复的状态映射代码

**现状**：`DashboardScreen.statusInfo()` 和 `TaskListScreen.statusLabelAndColor()` 是两个几乎相同的函数，各自维护一套 TaskStatus → (中文标签, 颜色) 的映射。

**建议**：在 `TaskStatus` 对象中直接提供 `label` 和 `color` 属性，或抽取为 `TaskStatusHelper` 单例，两处调用统一。

### 6. 控制面板显示全部任务 + 分页提示

**现状**：Dashboard 硬编码 `allTasks.take(10)`，超过 10 个任务时用户看不到完整列表，也没有"还有 N 个任务"的提示。

**建议**：改成全部显示（LazyColumn 已经高效），或显示前 10 条并加一条 "还有 N 个任务，点击查看全部 →" 的入口。

### 7. 操作反馈（Snackbar/Toast）

**现状**：清除日志、保存代理、启动/停止任务后无任何反馈提示，用户不确定操作是否生效。

**建议**：在关键操作后显示 Snackbar，例如"日志已清除""代理配置已保存""所有任务已启动"。

### 8. 日志自动清理

**现状**：日志表无限增长，只能手动"清除全部"。

**建议**：`LogEntryDao` 增加 `deleteOldest(keepCount: Int)` 方法，启动时自动保留最近 2000 条，超出部分静默删除。

### 9. 任务卡片增加搜索按钮

**现状**：任务页面的卡片没有查询按钮，只能回到控制面板手动输入 ID 查询。✅ **已在 1.1 版控制面板的 MiniTaskCard 中加入。**

**建议**：任务页面的 `TaskCard` 也加入同样的搜索图标按钮，无需切换页面即可查询。

### 10. 代理密码加密存储

**现状**：`ProxyManager` 用 SharedPreferences 明文存储代理密码。

**建议**：使用 AndroidX `EncryptedSharedPreferences` 或简单的 AES 加密存储密码字段。

---

## 三、低优先级（锦上添花）

### 11. 任务拖拽排序 / 置顶

**现状**：任务永远按创建时间倒序排列，用户无法调整顺序或置顶重要任务。

**建议**：任务表增加 `sortOrder` 字段，卡片支持长按拖拽重排（Compose `LazyColumn` + `detectDragGestures`）。

### 12. 首次使用引导

**现状**：新用户打开 App 面对空白界面，不知道从哪里开始。

**建议**：首次启动时显示 3~4 步的简单引导（高亮 FAB 创建任务 → 提示输入商品 ID → 启动 → 查看控制台），存标记到 SharedPreferences 不再重复。

### 13. Turbo 模式安全阀

**现状**：Turbo 模式零间隔连发请求，结合熔断器的 5 次阈值可能在熔断触发前就导致 IP 被限制。

**建议**：Turbo 模式下也保底 200ms 最小间隔，避免瞬间大量请求。

### 14. 退出 IP 查询降频

**现状**：每次 admire 请求成功后都调用 `api.ipify.org` 查询出口 IP，高频任务下产生大量不必要的查询。

**建议**：改为每 30 次成功请求或每 5 分钟查询一次出口 IP。

---

## 实施建议

如果按一个迭代排期，建议先做 **1~4**（自动重试 + 重新运行 + 日志过滤 + 查询走代理），这四项改动量小、独立性强，且直接解决用户最常遇到的痛点。5~10 可在后续版本分批合并。
